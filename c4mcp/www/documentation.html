C4 MCP Server

Installation & Configuration Guide

The C4 MCP Server driver exposes your Control4 system to AI assistants (Claude Code, Claude Desktop, OpenClaw, or any MCP-compatible client) over the local network. An LLM can query device state, control lights, thermostats, blinds, locks, and more – all through natural language.

Requires: Control4 OS 2.10.5+. HTTPS requires OS 3.3.0+ (TLS cert generation requires OS 3.4.1+).

Quick Start

  1. Install c4mcp.c4z on the controller via Composer Pro (search under Controllers).
  2. Go to the driver's Actions tab and press Generate API Key. Copy the key from the Last Generated Key property (it auto-clears after 30 seconds).
  3. Configure your MCP client to connect to http://<controller-ip>:9201/mcp with the API key as a Bearer token.

Claude Code – run this command:

claude mcp add --transport http --header "Authorization: Bearer <your-api-key>" c4home http://<controller-ip>:9201/mcp

Or add to ~/.claude.json (user scope) or .mcp.json (project scope):

{
  "mcpServers": {
    "c4home": {
      "type": "http",
      "url": "http://192.168.1.100:9201/mcp",
      "headers": {
        "Authorization": "Bearer <your-api-key>"
      }
    }
  }
}

You can now ask Claude questions like "What's the temperature?" or "Turn off the kitchen lights."

Authentication

The server uses Bearer token authentication. Every HTTP request must include an Authorization: Bearer <key> header.

Secure by default: If no API keys are configured, all requests are denied with HTTP 401. You must generate at least one key before the server will accept requests.

API keys are hashed (SHA-256) before storage – the plaintext key is only shown once at generation time.

Generating keys

In Composer Pro: Actions tab – Generate API Key – enter a name – key appears in the Last Generated Key property and Lua console output.

Revoking keys

Single key: Actions tab – Revoke API Key – enter the key name.
All keys: Actions tab – Revoke All API Keys.

HTTPS Configuration

Auto-generation (recommended)

  1. In Composer Pro, go to the driver's Actions tab.
  2. Press Generate TLS Certificate (OS 3.4.1+). This generates a self-signed certificate on the controller using lua-openssl (valid for 10 years).
  3. Set the Transport property to HTTPS.
  4. The Status property should show "HTTPS on port 9201".

Manual certificate upload

For CA-signed or custom certificates, use the Lua console in Composer Pro:

SetTLSCertificate(cert_pem, key_pem)

Both arguments accept full PEM format or raw base64 (headers are added automatically). There is no size limit when calling functions from the Lua console.

Print certificate

Use the Print TLS Certificate action to view the current certificate PEM in the Lua console output. This is useful for verifying the certificate identity on the client side.

Fallback behavior

If HTTPS is selected but no certificate is available or TLS server creation fails, the driver automatically falls back to HTTP and logs the reason. The Status property shows the error.

Trusting the Self-Signed Certificate

The Generate TLS Certificate action creates a self-signed certificate with the controller's IP address embedded as a Subject Alternative Name (SAN). This allows TLS clients to verify the server by IP address, which is required by modern TLS implementations including Node.js.

Step 1: Download the certificate

From a terminal on your workstation, use openssl s_client to download the certificate from the running server:

openssl s_client -connect 192.168.1.100:9201 -showcerts </dev/null 2>/dev/null \
  | openssl x509 -outform PEM > c4mcp.pem

Replace 192.168.1.100:9201 with your controller's IP and configured MCP port.

Step 2: Verify the certificate contains the controller IP

openssl x509 -in c4mcp.pem -noout -text | grep -A1 "Subject Alternative Name"

You should see output like:

X509v3 Subject Alternative Name:
    IP Address:192.168.1.100

If the SAN is missing (e.g. from an older certificate), regenerate it with the Generate TLS Certificate action.

Step 3: Trust the certificate

Option A: Claude Code / Node.js (recommended)

Set the NODE_EXTRA_CA_CERTS environment variable to point to the downloaded PEM file. Add this to your shell profile (~/.bashrc, ~/.zshrc, etc.):

export NODE_EXTRA_CA_CERTS=/path/to/c4mcp.pem

Then update your Claude Code MCP config to use https://:

{
  "mcpServers": {
    "c4home": {
      "type": "http",
      "url": "https://192.168.1.100:9201/mcp",
      "headers": {
        "Authorization": "Bearer <your-api-key>"
      }
    }
  }
}

Option B: System-wide trust (Linux)

sudo cp c4mcp.pem /usr/local/share/ca-certificates/c4mcp.crt
sudo update-ca-certificates

Option C: System-wide trust (macOS)

sudo security add-trusted-cert -d -r trustRoot \
  -k /Library/Keychains/System.keychain c4mcp.pem

Option D: System-wide trust (Windows)

Import c4mcp.pem via certmgr.msc: Trusted Root Certification Authorities – Certificates – right-click – All Tasks – Import.

Note: If the controller's IP address changes, you must regenerate the certificate (the SAN will no longer match) and re-download it.

Write Control

Controls which devices the MCP server can send commands to. Read operations (listing devices, reading state) are always allowed regardless of this setting.

Allow All (default)
All devices can be controlled.
Allowlist
Only devices selected in Write Control Devices can be controlled.
Blocklist
All devices except those selected in Write Control Devices can be controlled.

Use Allowlist mode to restrict the AI to specific devices (e.g. only lights). Use Blocklist to exclude sensitive devices (e.g. locks, security system).

MCP Tools

These are the tools the AI client discovers automatically via the MCP protocol. You do not need to configure these – they are built into the driver.

Read Tools

get_home
Home overview: room list, device count by type, number of lights currently on. The AI calls this first to orient itself.
get_devices
List devices with live state. Supports filtering by device_id (single device), room_id, or device_type. Use state_fields to limit which state fields are returned. Always includes controls.actions showing available commands for each device.
get_device_profiles
Show current Device Profiles configuration and count of built-in profiles.
configure_device_profiles
Returns a step-by-step prompt guiding the AI through Device Profile configuration.

Control Tools

control_device
Unified control for any device type. Parameters: device_id, action, and optional params. Actions and their parameter schemas are discovered by calling get_devices and reading the controls.actions field.
control_devices
Batch control. Accepts an array of {device_id, action, params} operations (max 20 per call). Returns per-device results with a summary.
set_device_profiles
Write new Device Profiles JSON configuration.

How control works

  1. The AI calls get_devices to discover devices and their available actions.
  2. Each device's state includes controls.actions – a map of action names to parameter schemas.
  3. The AI calls control_device with the device ID, action name, and required parameters.
  4. The driver validates the request (device exists, write access allowed, action valid, params in range) and dispatches the C4 command.
  5. The response confirms the command was dispatched. Commands are fire-and-forget – the AI can call get_devices again to verify the state changed.

Device Profiles

Device Profiles configure how custom devices (experience button proxies) and thermostats are exposed to the AI. Without a profile, custom devices default to a single "select" action with no context about what the device actually does.

Built-in Profiles

The driver includes profiles for common Control4 drivers, matched by driver filename:

Driver patternCategoryActions
lift_relay_control.c4zliftopen, close, stop
screen_relay_control.c4zscreenopen, close, stop
*_relay_control.c4zrelay_controlopen, close, stop
timer_*.c4ztimeron, off, set_countdown_mins
experience-button-scenario.c4zscenarioselect, set_state
uibutton_*.c4zav_sourceselect

User-configured profiles (by device ID) take priority over built-in profiles and fully replace them (no merging).

JSON Configuration Format

Set via the Device Profiles property or the set_device_profiles MCP tool. The value is a JSON object keyed by device ID (as strings):

{
  "243": {
    "category": "extract_fan",
    "states": {"0": "off", "1": "on"},
    "actions": {
      "on": "ON",
      "off": "OFF",
      "toggle": "TOGGLE",
      "set_countdown": "SetCountdown(Time[10,15,20,25,30,45,60]) @parent"
    },
    "extra_vars": ["COUNTDOWN", "RUNTIME"]
  }
}

Fields

category
Category label exposed to the AI (e.g. "extract_fan", "pool_heater").
states
Map of state values to labels: {"0": "off", "1": "on"} or {"Opened": "open", "Closed": "closed"}.
actions
Map of action names to compact command strings (see syntax below).
extra_vars
Array of extra variable names to include in device state (e.g. ["COUNTDOWN", "RUNTIME"]).
setpoint_type
Thermostats only: "single" (default), "heat", "cool", or "heat_cool".

Compact Action Syntax

SyntaxMeaning
"COMMAND"No parameters
"COMMAND(Param[A,B,C])"Enum parameter with allowed values
"COMMAND(Param[0-100])"Numeric range parameter
"COMMAND(Param)"Parameter accepting any value
"COMMAND(P1[a,b], P2[0-5])"Multiple parameters
"... @parent"Send command to parent device instead of proxy

Important: @parent – Custom devices are experience button proxies. Their commands (EX_CMDs) typically live on the parent driver, not the proxy. If a command goes to the wrong target, it is silently ignored. Use @parent for any command handled by the parent driver.

Thermostat Profiles

Thermostat profiles configure the setpoint type and can add custom actions beyond the standard set_mode/set_setpoint/set_fan/set_hold:

{
  "208": {
    "setpoint_type": "heat_cool",
    "actions": {
      "set_day_temp": "SET_DAY_TEMPERATURE(Degrees[5-30]) @parent",
      "set_setback_temp": "SET_SETBACK_TEMPERATURE(Degrees[5-30]) @parent"
    }
  }
}

Setpoint types:

"single" (default)
Single setpoint, sends SET_SETPOINT_SINGLE.
"heat"
Heat setpoint only, sends SET_SETPOINT_HEAT.
"cool"
Cool setpoint only, sends SET_SETPOINT_COOL.
"heat_cool"
Dual setpoints with heat/cool target selection.

Configuration via AI

The configure_device_profiles MCP tool provides a guided workflow. The AI will discover your devices, ask what commands each custom device supports, build the JSON, and apply it.

Lookup Order

  1. Device ID in user JSON config (highest priority).
  2. Driver filename in built-in profiles (exact match, then wildcard).
  3. Generic fallback (select only).

Finding Device IDs

Use the get_devices MCP tool, or in Composer Pro: System Design view, select a device, check the ID in properties.

Server Internals

Connection Management

  • Maximum concurrent connections enforced (default 5). Excess connections receive HTTP 429.
  • Idle connections are closed after the configured timeout (default 300 seconds).
  • Per-client rate limiting via sliding window (default 30 req/s). Excess requests receive HTTP 429.
  • A 60-second watchdog timer checks server health and restarts the listener if it goes down.
  • TCP server uses exponential backoff retry (3s–60s, up to 10 attempts) if the port is unavailable.

MCP Protocol

  • Implements MCP protocol version 2025-03-26 (Streamable HTTP transport).
  • Stateless – no session IDs, each request is self-contained.
  • POST /mcp is the only valid endpoint. Other paths return 404, other methods return 405.
  • JSON-RPC notifications (no id field) receive HTTP 202 with no response body.
  • Body limit: 64KB. Header limit: 8KB.

Troubleshooting

Server won't start / Status shows error

  • Check that port 9201 (or configured port) is not in use by another process.
  • Check the Lua console for error messages (enable Debug Mode for verbose output).
  • Use the Test MCP Response action to verify the MCP layer is working.

401 Unauthorized

  • Verify the API key is correct – keys are shown only once at generation.
  • Ensure the Authorization: Bearer <key> header is present in requests.
  • If no keys are configured, generate one first. The server denies all requests when no keys exist.

HTTPS not working

  • Check Status property for error details.
  • Verify a TLS certificate exists (use Print TLS Certificate action).
  • OS 3.3.0+ is required for TLS server support. OS 3.4.1+ is required for certificate auto-generation.
  • Use Generate TLS Certificate to create a new one, or SetTLSCertificate() from the Lua console for custom certs.

Devices not showing up

  • Use Refresh Device Cache action after adding or removing devices in Composer Pro.
  • The device cache is rebuilt automatically at driver startup.
  • Only supported proxy types are exposed: light, thermostat, lock, blind, relay, sensor, security, camera, door station, and experience button (uibutton).

Custom device shows only "select" action

  • The device needs a Device Profile. See the Device Profiles section above.
  • Check if a built-in profile already covers it (matched by driver filename).
  • Use the configure_device_profiles MCP tool to create a profile interactively with the AI.

Commands dispatched but device doesn't respond

  • control_device confirms dispatch, not execution – commands are fire-and-forget.
  • Check Write Control Mode – the device may be blocked.
  • For custom devices, ensure the action uses @parent if the command is handled by the parent driver.
  • Verify the command name and parameters match what the device actually supports.

Connection drops / "Too many connections"

  • Default max is 5 simultaneous connections – increase via Max Connections if needed.
  • Idle connections are closed after 300 seconds – increase Idle Timeout if the client reconnects frequently.

Rate limiting (429 errors)

  • Default is 30 requests/second per client.
  • Increase via Max Requests/sec if the AI is making rapid successive calls.
